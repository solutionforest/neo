Core workflow

Four steps from bare VPS to live app.

Every operation goes through SSH. No agent installed, no control plane to maintain. Neo reads your existing docker-compose.yml and .env files automatically.

1. Initialize the machine

Connect over SSH, verify the OS (Ubuntu 24.04+ or Debian), install Docker, create the neo network, start Caddy, and persist server state.

2. Install or deploy

Use a built-in template for a known stack (Ghost, Plausible, WordPress...), or deploy your own Dockerfile-based app directly.

3. Route traffic

Caddy stays in front of the apps so domains and TLS certificates remain part of the same operating flow. Zero-downtime config reloads.

4. Keep operating locally

Logs, status checks, restarts, domain changes, and follow-up commands still run from your machine through the same SSH tunnel.

Commands

Every command Neo provides.

All commands use the same pattern: resolve the server, connect via SSH, operate, and save state.

neo
Launch the interactive TUI dashboard. Browse servers, apps, and actions without memorizing flags.
neo init <host>
Initialize a new server. Validates the OS (Ubuntu 24.04+ or Debian required), installs Docker, starts Caddy, creates the neo network, and saves server state.
neo install <app>
Install a templated app (Ghost, Plausible, WordPress, etc.) with bundled database, volumes, and routing.
neo deploy .
Build from a local Dockerfile, transfer the image via SSH, start the container, and configure routing. Auto-reads env vars from docker-compose.yml, .env, or .neo.yml. Deploys background workers and persistent volumes declared in .neo.yml. Supports --env KEY=VAL, --env-file .env, and --to <env> flags.
neo env <app>
View all environment variables for an app. Secrets are automatically masked in output.
neo env set <app> K=V
Set or update environment variables. The container restarts automatically with the new config.
neo env unset <app> KEY
Remove an environment variable. The container restarts automatically with the updated config.
neo env import <app> .env
Bulk import variables from a .env file. Existing values are updated, new ones are added.
neo list
List all running apps on the current server with their status, domains, workers, and ports.
neo logs <app>
Stream live container logs from the remote server. Supports --tail, --follow, and --worker <name> flags.
neo domain <app> <domain>
Assign or update a domain for a running app. Caddy provisions the TLS certificate automatically.
neo stop|start|restart <app>
Control the lifecycle of an app and its workers on the remote server. All worker containers are started, stopped, or restarted alongside the main app.
neo update <app>
Pull the latest image for an app and redeploy it. Useful for keeping templated apps up to date.
neo remove <app>
Stop and remove an app container, its services, and its Caddy routing entry.
neo servers
List all configured servers. Add, switch, or remove servers from the local config.
neo use <name>
Switch the active server. All subsequent commands target this server unless overridden with --server.
neo ssh
Open an interactive SSH session to the current server. Replaces the process on Unix systems.
neo version
Show the current version and check for updates. Contacts the download server to compare versions.
neo upgrade
Self-update neo to the latest version. Downloads the correct binary for your OS/arch and replaces the current installation in-place.
neo backup <app>
Backup an app's data volumes to a compressed archive on the server. Optionally download to your local machine. NEO+
neo restore <app> <file>
Restore an app from a backup archive. Overwrites current data volumes with the backup contents. NEO+
neo volumes
List all Docker volumes on the current server, showing which app owns each volume.
neo volumes mount <vol> <path>
Mount a Docker volume to a host path (e.g. external SSD). Briefly stops the app, copies data, and remounts. NEO+
neo connect
Migrate the current server to Vxero. One-way transfer — your Docker apps keep running while Vxero adds the management layer.
Configuration

.neo.yml reference

Optional project config file. All fields are optional — Neo works fine without it.

# App identity
name: my-app              # app name (default: directory name)
domain: app.example.com   # domain (default: prompt or nip.io)
port: 8080                # container port (default: Dockerfile EXPOSE)
server: production        # pin to a named server

# Environment variables (non-sensitive defaults)
env:
  APP_ENV: production
  LOG_CHANNEL: stderr
env_file: .env.production  # load from file

# Persistent volumes (survive redeploys)
volumes:
  data:
    path: /var/www/html/database
  storage:
    path: /var/www/html/storage

# Background workers (same image, different command)
workers:
  queue:
    command: php artisan queue:work --tries=3
  scheduler:
    command: php artisan schedule:work

# Named environments (override top-level values)
environments:
  production:
    domain: app.example.com
  staging:
    domain: staging.example.com
    env:
      APP_DEBUG: "true"
workers
Each worker runs as a separate container using the same image, env vars, and volumes as the main app — but with a custom command. Workers are deployed with blue-green rollout, managed by start/stop/restart, and removable by deleting them from the config. View logs with neo logs <app> --worker <name>.
volumes
Named Docker volumes are created on first deploy (prefixed with the app name, e.g. my-app-data) and preserved across all subsequent redeploys. New volumes added to .neo.yml are created on the next deploy without affecting existing ones.
environments
Named deployment targets that override top-level config. Deploy to a specific environment with neo deploy --to staging. If multiple environments exist and no --to flag is provided, Neo prompts you to select one.

Server Requirements
Neo requires Ubuntu 24.04+ or Debian on the target server. Older Ubuntu versions (22.04, 20.04, etc.) and other distros (CentOS, RHEL, Alpine, Fedora) are not supported. neo init will automatically detect your OS and display an error if it is unsupported.

Neo connect

One command to upgrade to Vxero.

When you outgrow the CLI, neo connect migrates your server to the full Vxero platform. Your Docker apps keep running — you just get a management layer on top.

How it works

The transfer is one-way. Neo analyzes your apps and services, creates a migration plan, and hands the server to Vxero.

$ neo connect Analyzing server state... Found 3 apps, 2 services (mysql, redis) Migration plan: ghost -> Vxero project (Docker) plausible -> Vxero project (Docker) mysql -> Managed MySQL service redis -> Managed Redis service Connecting to Vxero... Server transferred successfully. Dashboard: https://app.vxero.dev/servers/prod-1

Zero downtime

Your apps keep running in Docker while Vxero gradually transitions management. No restart, no migration window.

Team access unlocked

After migration, invite teammates with role-based access, monitoring dashboards, and audit logs — features a local CLI can't provide.
